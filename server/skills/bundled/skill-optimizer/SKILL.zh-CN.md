# 技能优化器

## 使命

把真实使用证据转化为更安全、更可靠、更容易触发、更容易评估的 Agent Skill。

这是一个元技能。它不是简单改写文字，而是分析技能的目标、触发描述、输入、流程、工具、输出、风险、示例和评估方式之间是否匹配，然后提出小而可审查的修改。

## 使用场景

当用户要求改进、优化、调试、重构、基准测试、验证、泛化、加固、打包或记录另一个技能时使用。

也适用于：

- 当前对话应该沉淀成技能。
- 上一次技能运行暴露出问题，需要下次避免。
- eval 失败说明技能触发或执行方式有缺陷。
- 用户纠正了 agent 的行为，需要把纠正固化到技能中。

不要用该技能执行目标技能本身的普通任务。如果用户要求运行发布技能，就运行发布技能；只有当用户要求改进发布技能时，才使用本技能。

## 模式

- Diagnose：只诊断应该改什么，不直接写 patch。
- Patch：生成可审查 diff、替换章节或新版 `SKILL.md`。
- Validate：创建验证用例和失败模式检查。
- Benchmark：用任务案例、触发案例和确定性评分比较新旧技能。
- Package：整理技能目录、参考资料、脚本、示例、变更记录和 README。

## 必要输入

在提问前尽量从对话中推断：

- 目标技能名称和用途。
- 当前 `SKILL.md` 和支持文件。
- 执行证据：用户请求、工具使用、输出、纠正、错误、延迟或意外行为。
- 运行环境：chat、coding agent、workspace agent、API 或 harness。
- 期望优化模式。
- 风险级别和写入权限。
- 如果优化发生在另一个实时任务之后，需要确认执行顺序约束。

如果无法访问目标技能文件，不要伪造精确 diff。应输出基于假设的改进计划、替换章节草稿、验证用例和明确假设。

## 工作原则

- 以证据驱动修改，不凭感觉重写。
- 优先小 patch，避免大范围改写导致不可审查。
- 先修触发条件、执行步骤、失败模式和验证方式这些高影响部分。
- 不要把当前一次性上下文硬编码为长期规则。
- 每次优化都要说明预期收益和可能风险。
