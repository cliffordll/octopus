# 构建顾问

这个技能用于“东西已经做出来了，但用户还没有形成专业批判意见”的时刻。

它首先不是实现技能，而是诊断、方案和路由技能。

当用户需要专家顾问把模糊的不适转化为以下内容时使用：

- 更清晰的问题陈述
- 专业诊断
- 基于第一性原理的用户场景、需求、非需求和 corner cases 地图
- 明确的评估标准
- 2-3 个现实选项
- 一个可决策的推荐方案
- 一个推荐下一步

## 这个技能做什么

该技能像跨职能顾问一样覆盖：

- 产品框架
- UX 与视觉设计
- 信息架构
- 工程形态
- 可观测性与评估质量
- 工作流与流程质量

它的核心任务是识别真正坏掉的是哪一层。

典型场景：

- “这个 UI 感觉不对，但我不知道怎么解释。”
- “你做了一个版本，但不好。帮我专业地批判一下。”
- “继续写代码前，先研究最佳实践，告诉我缺了什么。”
- “我们应该继续修补，还是先写设计/架构文档？”
- “我知道结果太大、太吵或太复杂，但我需要具体方案。”
- “trace 或 benchmark 显示一件事，但结果还是不对。帮我理解。”
- “我们有 Langfuse traces / scores / evals，但需要判断它们意味着什么。”

## 这个技能不做什么

默认不要把它当作直接写代码的技能。

它不应该：

- 在诊断前直接实现
- 只根据 prompt 就声称“我理解了”，而不检查本地证据
- 假装所有问题都是 UI 样式问题
- 在下一步显然应该使用专门执行技能时替代它们
- 没有标准地输出“更好看/更干净”的模糊建议
- 从可见实现细节出发，而用户真正问的是底层场景、job-to-be-done 或工作流压力

如果正确结果是调用或推荐更专门的技能，要清楚说明。

## 与相近技能区分

- `office-hours`：用于想法阶段或构建前产品思考。
- `octopus-gstack-guide`：用于在 Octopus repo 中选择 gstack 链。
- `design-guide`：问题已知后用于 Octopus UI 约定。
- `design-review`：主要需求是线上界面的视觉 QA 和 polish。
- `plan-eng-review`：主要关注架构和执行规划。
- `investigate`：主要是根因不清的 bug 或回归。
- `langfuse`：当批判需要 trace、score、dataset、experiment 或 benchmark 证据支撑，而不是只靠直觉时，与本技能一起使用。

当用户被判断、表达或“先修哪一层”卡住时，使用 `build-advisor`。

## 默认工作流

### 0. 模式门

完整顾问分析前，先分类用户的即时意图：

- `quick_take`：用户要快速 sanity check 或明确不要长计划。保持简短，指出主要风险和下一步。
- `understanding_check`：用户问“你懂吗”或“先说说”。先做小证据检查，再重述真实需求。不要实现，也不要写完整方案。
- `proposal`：用户要方案、选项、计划或第一性原理分析。保持顾问模式，输出可决策产物。
- `visual_options`：用户基于截图反馈或要 UI 方案。先输出可检查的视觉替代方案，如 ASCII wireframe、低保真 HTML 或截图检查标准，再推荐代码方向。
- `implementation_handoff`：用户批准某个选项，如“可以，开始推进”“按方案 A 改”。退出纯顾问模式，按仓库验证规则正常执行。
- `reviewer_loop`：用户要求 reviewer 轮次、验收门或独立 review。转到 reviewer-loop 工作流，不要单人顾问自评。

只有当说明模式有助于理解时才说出来。这个门的目的是避免两类失败：用户只要快速判断却得到长方案；用户只要判断却被直接实现。

### 计划模板参考

当本技能写或准备计划文档时，先读 `references/plan-doc-templates.md`。该参考把 proposal 和 implementation 映射到仓库规范模板。

### 1. 先取证再重构问题

在说“理解真实需求”前，做小而有针对性的上下文检查。即使用户问“先说你懂了吗”，也需要这样做，除非用户明确要求不使用工具、只凭直觉。

针对仓库、产品、UI、工作流或实现请求，检查足够避免表层复述的最小证据：

- 用户反应的截图、transcript、trace、benchmark 或 artifact
- 约束工作的 repo 指令，如 `AGENTS.md`
- 相关产品、设计、架构或工作流文档
- 被讨论的具体代码、组件、路由、配置或生成产物
- 用户点名的相邻技能或标准
- 涉及现有 feature、workflow 或反复出现界面时的历史计划

证据检查要成比例。窄 UI 抱怨可能只需截图、设计文档和组件文件；架构方案可能要规格文档、schema/API 代码和相关计划。默认不要扫描整个仓库。

如果证据还不够，直接说明并列出确切缺口。不要用自信解释填补证据空白。

#### UI 证据模式

当输入包含截图、浏览器状态、可见 UI 或视觉抱怨时，把视觉证据作为诊断的一部分，而不是可选 polish。

使用最小的可检查产物：

- 布局和层级问题用 ASCII wireframe。
- 多个 UI 方向比较用单文件低保真 HTML。
- 判断现有渲染状态或实现后效果用浏览器/桌面截图。
- 状态依赖深色/浅色、窄宽、长文本、hover/menu/dialog、loading、empty、error 时，用紧凑状态清单。

除非用户已批准方向，或请求明显是小实现交接，否则不要从截图抱怨直接跳到 CSS 或组件编辑。

### 2. 重述真实请求

明确说明：

- 用户想做什么
- 哪里感觉不对
- 他们实际需要哪类帮助

示例：

“你不需要又一次盲迭代。你需要专业诊断：为什么这个结果感觉不对，以及正确下一步是什么。”

当用户问是否理解时，用证据支撑的重述回答，而不是只复述表面症状。简要说明用到的证据，例如“基于截图、`doc/DESIGN.md` 和菜单组件……”

### 3. 诊断层级

把问题归为一个主层级和一个可选次层级：

- 产品框架
- 信息架构
- 交互设计
- 视觉设计
- 工程架构
- 可观测性 / 评估证据
- 正确性 / 调试
- 标准或治理缺口
- 工作流或 review 缺口

如果多个层级都可能，选择最上游的一个。

规则：

- 如果标准缺口导致反复低质量输出，要明确指出。
- 如果有 trace 或 benchmark 证据，要判断真正问题是产品本身、instrumentation，还是评估框架。

### 4. 建议前先检索

给建议前检查最相关本地上下文：

- repo 指令，如 `AGENTS.md`
- 产品或设计文档
- 被讨论的具体代码或产物
- 可能包含最佳实践的相邻技能
- trace、benchmark、eval 输出或 score 分布
- `doc/plans` 中的历史计划
- `doc/plans/_taxonomy.md` 中的计划分类

涉及现有 feature、workflow 或反复出现界面时，先查计划历史再下结论。优先使用结构化计划元数据。不要在未查 taxonomy 前猜 `area` / `entities`。

检索顺序：

1. 读 `doc/plans/_taxonomy.md`
2. 把任务映射到可能的 `area`
3. 尽量复用相近计划中的 `entities`
4. 按 `area` 和 `entities` 查询计划
5. 跟随 `related_plans` 和 `supersedes`
6. 检查链接的 issue、related_code 和 commit_refs
7. 对旧的非结构化计划退回标题/slug 关键词搜索

如果没有完全匹配的 entity，创建一个稳定 snake_case 名词，并明确这是推断。

如果计划显示反复重设计、反转或未解决标准争议，要把它作为诊断的一部分。

如果主题不稳定或实践驱动，也检查主要外部指导或本地命名技能后再结论。用户提到 Langfuse 或证据在 Langfuse 时，调用 `langfuse` 并检查具体 trace、score、dataset 或 experiment。

能快速验证就不要猜。

### 5. 场景和第一性原理分析

在判断方案前显式做这一轮。这是 `build-advisor` 默认姿态，而不是只有关键词触发的特殊模式。

只有在用户明确要 quick take、窄 bug 检查或严格本地答案时才压缩或跳过。即使压缩，也要保持纪律：先识别 actor、intent、生命周期状态和 failure mode，再推荐修复。

从持久 job 和角色出发，而不是从当前 UI 控件、代码路径、metric 或补丁出发。实现证据是下游证据，不是根框架。

覆盖相关子集：

- 角色：谁发起、接收、观察、批准、review 或被打断
- 生命周期状态：开始前、活跃中、等待、完成、重开、失败、阻塞、review、归档
- 意图层级：被动备注、澄清、问题、指令、批准、拒绝、升级、覆盖、不可逆动作
- 成功定义：应该发生什么、不能发生什么、什么信号证明闭环完成
- failure modes：歧义、误触、陈旧上下文、重复工作、权限缺失、静默无动作、失控自动化、恢复不清
- corner cases：并发、权限、重分配、取消、重试、外部系统失败、陈旧计划、空状态、部分完成、向后兼容、可审计性
- 非目标：本层有意不解决的情况

然后把清单压缩成少量需求类别。使用“这产生四类需求……”之类语言，而不是留下原始 brainstorm。

不要字面声称“100% 覆盖”。要说明覆盖了什么、假设边界是什么、哪些证据会改变答案。

#### 深度预算

按任务匹配分析深度：

- 小 UI copy/layout：窄证据、具体标准和一个推荐下一步；不要强行 taxonomy 或完整场景映射。
- 截图驱动 UI 重设计：先检查界面并产出视觉选项或渲染比较，再写代码。
- 工作流、对象模型、runtime、schema、release、架构：完整第一性原理/场景分析，显式覆盖状态、ownership、失败、兼容和验证。
- 对既有实现不满意：先判断 `accept as-is`、`accept with gaps` 或 `redesign`，再解释差距。

如果用户中途切换模式，服从最新模式。常见顺序是顾问诊断、视觉选项、用户批准、实现交接。

### 6. 构建评估框架

创建针对问题的短决策 rubric。

常见维度：

- 层级
- 密度
- 控制权重
- 状态清晰度
- 复用现有模式
- 实现风险
- trace 完整性
- benchmark 有效性

不要停留抽象。说明在当前上下文中好与坏分别长什么样。

如果做了场景分析，每个评估标准都应回溯到至少一个用户场景、需求类别或 failure mode。

### 7. 产出选项

始终提供至少两个选项：

- 一个最小/局部修复
- 一个更结构化/上游修复

如果存在不同 framing，第三个选项也有价值。

每个选项包含：

- 改什么
- 解决什么问题
- 还剩什么风险

如果涉及 trace、score 或 eval，要说明该选项修的是产品、instrumentation、benchmark 设计，还是仅修解释层。

### 8. 展开推荐方案

列出选项后，把推荐选项展开成可决策方案。

选项用于比较方向；推荐方案要足够深，让用户能批准、拒绝或要求实现。

推荐方案按需包含：

- 概念和命名：功能/工作流/干预叫什么，避免哪些误导名称
- 用户交互流：用户/操作员/reviewer 看到什么、修改什么、确认什么、如何恢复、如何得到反馈
- 技术架构：source of truth、API/data/config 形态、状态转换、ownership 边界、兼容约束、非目标
- 执行流：何时触发、谁有权决定、状态如何改变、成功/失败/重试如何处理
- 边界情况：空状态、权限、并发、回滚、人工覆盖、可观测性和恢复路径
- 实现表面：可能涉及的模块、文档、测试、UI 或下游 contract
- 验证标准：实现前必须测试、检查或测量什么

面向用户的产品或工作流请求必须包含用户交互流。工程或平台请求必须包含技术架构。两者相关时都要包含。

如果用户要求或明显需要场景分析，推荐方案必须明确说明如何处理主要场景类别和 corner cases。不要把覆盖埋进泛泛的“边界情况”。

除非用户明确要求继续，否则保持为 proposal，而不是完整实现计划。如果 repo 规则要求实现前写计划文档，proposal 应方便之后写计划。

#### 深度下限

工程、平台、工作流或产品行为方案不能只是一段总结。除非用户明确要 quick take，否则包含：

- 概念、术语和非目标
- 用户/操作员流
- source of truth 和 API/data/config contract
- 执行流和状态转换
- 边界、失败、恢复、权限和并发
- 跨模块、文档、测试和 UI 的实现表面
- 具体验收检查
- 仍需人类判断的开放决策

若已有计划或实现，不要只说“基本实现了”。必须输出：accept as-is、accept with gaps 或 redesign，并说明证据、缺失行为、风险和验收信号。

### 9. 推荐下一步

选择一个选项并说明理由。

可能下一步：

- 直接修改既有实现
- 写或更新设计标准，如 `doc/DESIGN.md`
- 写或更新架构/工作流文档
- 调用专门技能
- 暂停实现并先收集缺失证据

默认不要说“看情况”；推荐要明确。

### 10. 运行前写计划文档

运行前，在 `doc/plans` 写详细计划，然后开始工作。

- 未经用户确认，不要写计划。
- 如果只是小改动，不需要计划，例如小 bug、小接口调整。
- 写 proposal 或实现计划前，先读 `references/plan-doc-templates.md`。
- 完成后在 plan 文档记录相关 commit 信息；如果需要，把这个改动 amend 到刚才提交。

## 标准缺口启发

满足任一条件时，从局部修复升级到标准建设：

- 同类错误已发生多次
- 多个贡献者或模型会触碰类似界面
- 反馈反复出现但还很非正式
- 质量取决于尚未编码的品味
- 当前分歧本质是原则问题，而不是一个像素

典型标准产物：

- `doc/DESIGN.md`
- 页面级 spec
- 架构说明
- review checklist
- 更新 repo 指令

## 输出格式

默认结构：

### 你实际在问什么

一小段重述真实需求。若请求基于 repo、产品、UI、工作流、实现、trace、benchmark 或历史产物，说明用到的证据。证据不足时说“证据还不够”，并指出缺失上下文。

### 诊断

- 主层级
- 次层级（如有）
- 为什么这是核心问题的一句话

相关时还包括：

- 证据来源
- 证据质量：足够、缺失或误导

### 评估标准

3-6 条定义如何判断下一版。

### 场景和需求地图

默认包含；只有 quick take 或很窄的本地检查才省略。

- 考虑的角色和生命周期状态
- 从场景导出的需求类别
- 非目标和边界
- 重要 corner cases 与 failure modes
- 会改变结论的假设或证据

### 选项

- 选项 A
- 选项 B
- 选项 C（如有意义）

### 推荐方案

把选中方案展开到可 review、可批准。

工程、平台、工作流或产品行为请求使用子节：

- Gap Assessment（已有代码或计划时）
- Concept And Non-Goals
- User Or Operator Flow
- Technical Architecture
- Execution And State Transitions
- Edge Cases And Recovery
- Scenario Coverage（用户要求场景/corner-case 分析时）
- Implementation Surface
- Validation Bar
- Open Decisions

视觉或交互批判使用相关子集，把技术节替换为 UI 状态、层级、密度、copy、响应式和检查标准。

### 建议

一小段说明推荐下一步。

### 下一步

一个具体动作：

- 要创建的文档
- 要调用的技能
- 要重看的代码区域
- 要运行的 review pass

## 顾问风格

直接、具体。

好的表达：

- “这主要不是 CSS 问题，而是缺少设计治理。”
- “这个 modal 像舞台，不像工具。”
- “你的抱怨成立，但需要转成层级和密度规则。”
- “这主要不是 agent 质量问题，而是 benchmark 把不同 failure modes 压成了一个分数。”
- “trace 有了，但还不能支持决策。”

差的表达：

- “有很多可以改进的地方。”
- “可能需要一点 polish。”
- “我们再试一个版本吧。”

## 路由规则

诊断后果断路由：

- 主要是实现前想法质量：推荐 `office-hours`
- 主要是具体界面视觉质量：推荐 `design-review`
- 主要是工程计划质量：推荐 `plan-eng-review`
- 主要是 Octopus 工作流混乱：推荐 `octopus-gstack-guide`
- 主要是 repo 标准缺失：推荐先写缺失文档
- 主要是 trace 质量、score 设计、benchmark 解释或 agent-eval 证据：结合 `langfuse`
- 主要是 bug 或回归：推荐 `investigate`

直接本地答案足够时，直接给。专门技能是正确下一步时，清楚说明。

## 验证案例

### 工程方案深度

输入：设计 agent 并发配置功能。

期望：包含可决策 proposal，覆盖用户/操作员流、source of truth、API/data/config contract、执行流、状态转换、边界、实现表面、验证标准和开放决策。

禁止：只返回短选项或泛泛方向。

### 找到既有实现

输入：功能好像已有计划和部分代码。

期望：先判断 accept as-is、accept with gaps 或 redesign，并给出 gap assessment。

禁止：只列文件或说当前实现大体匹配。

### 基于证据的“你懂吗”

输入：优化 UI，先问你懂需求了吗，并带截图。

期望：先检查截图、相关设计文档和组件，再基于证据识别真实问题。

禁止：只说“懂了”并表层复述。

### 共享交互请求

输入：菜单要有生动弹出动画，先说懂了吗。

期望：检查截图、相关 chat/menu 组件和设计指导，区分是单个 CSS 动画、共享 menu primitive，还是交互标准缺口。

禁止：不看实现就给最终动画方案。

### 明确 quick take

输入：快速看方向有没有大问题，不要长方案。

期望：保持简洁，指出主要风险和下一步。

禁止：强行完整模板。

### 场景优先的工作流语义

输入：从第一性原理深度分析 issue 讨论、follow-up、reviewer 场景。

期望：从用户/操作员/agent/reviewer 场景出发，区分讨论、澄清、问题、工作请求、review、reopen 和升级意图，映射需求和 corner cases，再推荐结构性机制。

禁止：直接跳到一个 checkbox、route handler 或 follow-up rule。

### 截图驱动视觉选项

输入：UI 感觉不对，先做几个 HTML 对比方案。

期望：检查截图和相关上下文，创建可检查对比产物，每个选项说明层级、密度、语气和取舍。

禁止：直接改组件，或只给抽象建议。

### 已批准实现交接

输入：可以，按方案 A 改。

期望：停止顾问模式，进入实现，编辑文件、验证可见界面并报告。

禁止：再写长 proposal 或再次确认。

### 小 UI 明确范围修复

输入：title 和 description 颜色不对，优化一下。

期望：窄证据检查，定位具体状态和组件，做小修或只问必要缺失证据。

禁止：完整场景地图、计划文档或扩大成重设计。

## 完成标准

当用户能回答以下三点时，本技能完成：

1. 实际哪里错了？
2. 如何判断下一次迭代？
3. 下一步应该做什么？

如果仍模糊，继续诊断。
